198 /*S
199  MPID_Info - Structure of an MPID info
200 
201  Notes:
202  There is no reference count because 'MPI_Info' values, unlike other MPI
203  objects, may be changed after they are passed to a routine without
204  changing the routine''s behavior. In other words, any routine that uses
205  an 'MPI_Info' object must make a copy or otherwise act on any info value
206  that it needs.
207 
208  A linked list is used because the typical 'MPI_Info' list will be short
209  and a simple linked list is easy to implement and to maintain. Similarly,
210  a single structure rather than separate header and element structures are
211  defined for simplicity. No separate thread lock is provided because
212  info routines are not performance critical; they may use the single
213  critical section lock in the 'MPIR_Process' structure when they need a
214  thread lock.
215 
216  This particular form of linked list (in particular, with this particular
217  choice of the first two members) is used because it allows us to use
218  the same routines to manage this list as are used to manage the
219  list of free objects (in the file 'src/util/mem/handlemem.c'). In
220  particular, if lock-free routines for updating a linked list are
221  provided, they can be used for managing the 'MPID_Info' structure as well.
222 
223  The MPI standard requires that keys can be no less that 32 characters and
224  no more than 255 characters. There is no mandated limit on the size
225  of values.
226 
227  Module:
228  Info-DS
229  S*/
230 typedef struct MPID_Info {
231  int handle;
232  int pobj_mutex;
233  int ref_count;
234  struct MPID_Info* next;
235  char* key;
236  char* value;
237 } MPID_Info;
